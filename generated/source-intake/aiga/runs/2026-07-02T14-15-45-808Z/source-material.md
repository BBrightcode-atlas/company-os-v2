# 0. 사용자 등급 정의

> 💡 AIGA의 모든 정책은 비회원 / 일반 회원 / 의사 인증 회원 3단계를 기준으로 합니다. 공통 정책은 한 번만 정의하고, 각 등급은 공통과 다른 부분만 추가 정의합니다.

| 등급 | 정의 | 특징 |
| --- | --- | --- |
| 비회원 | 로그인 없이 접속한 사용자 | 열람 일부 제한, 작성 불가 |
| 일반 회원 | 회원가입 완료한 사용자 | 게시글/리뷰 작성 가능, 인증 취득 가능 |
| 의사 인증 회원 | 회원가입 + 의사 면허 인증 완료 | 전문가 뱃지, 리뷰 작성 가능(본인 프로필 제외) |

# 1. 게시글 정책

## 1-1. 등급별 권한

| 기능 | 비회원 | 일반 회원 | 의사 인증 회원 |
| --- | --- | --- | --- |
| 게시글 열람 | 일일 N회 제한(데모에선 5회) | 무제한 | 무제한 |
| 게시글 작성 | 불가 (→ 1-1-1) | 가능 | 가능 |
| 게시글 수정 | 불가 (→ 1-1-1) | 본인 글만 | 본인 글만 |
| 게시글 삭제 | 불가 (→ 1-1-1) | 본인 글만 | 본인 글만 |
| 공감 | 불가 (→ 1-1-1) | 가능 | 가능 |
| 댓글 | 불가 (→ 1-1-1) | 가능 | 가능 + 의사 뱃지 표시 |
| 병원 방문 인증 | 불가 (→ 1-1-1) | 선택 가능 | 선택 가능 (환자 입장으로 인증) |
| 게시글 뱃지 | 없음 | 인증 시 병원 진료 인증 뱃지 | 의사 전문가 뱃지 |

### 1-1-1. 비회원 권한 제한 시 공통 UX

- 비회원이 공감 · 댓글 · 리뷰 쓰기 · 즐겨찾기 · 신고 등 권한 없는 기능을 시도할 경우, 공통 컴포넌트인 LoginRequiredToast(중앙 모달형)를 노출한다.
- 이 규칙은 게시글 · 리뷰 · 의사 프로필 등 모든 영역에 동일하게 적용되며, 개별 화면정의서에서 별도 정의하지 않는다.
### 1-1-2. 비회원 게시글 열람 제한 (상세 진입 기준)

- “열람”은 커뮤니티 게시글 ‘상세 화면(게시글 상세 모달)’ 진입을 의미한다. 목록(피드)에서의 카드 노출/스크롤은 열람 횟수에 포함하지 않는다.
- 열람 횟수는 상세 화면 진입 시점에 1회 차감한다.
- 동일 게시글을 같은 날 재진입하더라도 ‘진입’ 기준으로 차감한다(=재진입도 1회로 계산). 단, 상세 화면 내에서 새로고침/뒤로가기 등으로 동일 상세가 즉시 재로딩되는 경우는 중복 차감을 방지한다(세션/쿨타임 기준으로 1회만 차감).
- 잔여 횟수 N회 이하부터 UsageLimitBanner로 경고한다. (비회원 일일 열람 한도 N값은 발주사와 협의 후 확정)
- 잔여 횟수 0회가 되면 비회원은 상세 진입이 차단되며, 로그인 유도 UI(LoginRequiredToast)를 노출한다. 목록/검색/필터 등 탐색 기능은 유지한다.
### 1-1-3. 작성 정책 (컨텐츠 유형별)

| 컨텐츠 유형 | 본문 글자 수 | 이미지 첨부 | 비고 |
| --- | --- | --- | --- |
| 게시글 제목 | 최대 50자 | — | 피드 카드에서 1줄 초과 시 … 말줄임 처리 |
| 게시글 본문 | 무제한 | 최대 10장 (JPG·PNG·WEBP, 장당 5MB) | 텍스트 + 이미지 통합 에디터. 피드 카드에서 최대 2줄 미리보기 |
| 리뷰 | 최소 50자 / 최대 무제한 | 최대 10장 (JPG·PNG·WEBP, 장당 5MB) | 별점 필수 입력 |
| 댓글 | 1,000자 제한 | 불가 (텍스트만) |   |
| 대댓글 | 1,000자 제한 | 불가 (텍스트만) |   |

## 1-2. 수정 정책

> ⚠️ 수정 기간: 어드민 그룹별 N일 설정으로 운영 중 변경 가능 — §5-0 수정·삭제 기간 운영 정책 참조. 수정 이력 표시는 변경 여부와 무관하게 적용합니다.

### 1-2-A. 게시글 수정

| 항목 | 정책 |
| --- | --- |
| 수정 기간 | → §5-0 수정·삭제 기간 운영 정책 참조 (어드민 그룹별 N일 설정) |
| 수정 이력 표시 | 수정 시 '수정됨' 표시. 목록 카드와 상세 화면에 동일하게 노출하며, 호버 시 수정 일시 툴팁 제공 |
| "수정" 기준 | 본문/제목/첨부 이미지/태그(질환명 등) 변경은 모두 "수정"으로 간주하여 '수정됨'을 표시한다. 단, 병원 방문 인증값 변경(추가/초기화 등)은 "수정"에 포함하되, 인증 정책에 따라 별도 상태 변경(예: 인증 초기화)이 우선 적용된다. |
| 인증 상태 - 내용 수정 | 인증 유지 |
| 인증 상태 - 질환명 변경 | 인증 초기화, 재인증 필요 안내. 단, "변경"의 기준은 (자유 텍스트가 아닌) 구조화된 입력값 변경으로 정의한다: 질환명(태그/선택값) 변경 시에만 인증을 초기화한다. 본문 내 자유 텍스트에서 질환명이 바뀌는 경우는 시스템이 변경 여부를 판별할 수 없으므로 인증 초기화 대상에 포함하지 않는다. |
| 미인증 게시글에 인증 추가 | 수정 화면에서 언제든 병원 방문 인증을 추가할 수 있음 |
| 인증된 게시글 재인증 | 기존 인증 초기화 후 새 영수증 업로드. OCR 자동 인식 성공 시 즉시 인증 뱃지 재부여. 인식 실패 시 인증 미부여 + §3-1 "OCR 실패 안내 팝업" 표시 |

> ⚠️ 삭제 기간: 어드민 그룹별 N일 설정으로 운영 중 변경 가능 — §5-0 수정·삭제 기간 운영 정책 참조. 댓글이 있는 글 삭제 시 확인 팝업은 항상 표시합니다.

### 1-3-A. 게시글 삭제

| --- | --- |

| 삭제 후 평점 | 의사 평점 즉시 재계산 |
| 댓글 있는 리뷰 삭제 | 확인 팝업: '댓글 N개가 함께 삭제됩니다. 삭제할까요?' → 동의 시 리뷰와 모든 하위 댓글·대댓글이 함께 화면에서 숨김 처리 (연쇄, 게시글과 동일 패턴). DB는 모두 soft delete 보관. |
| 댓글 개별 삭제 | 리뷰 내 댓글·대댓글의 개별 삭제는 §1-3-B 댓글/대댓글 삭제 정책을 그대로 적용 (의사 인증 회원 정의서 §6 리뷰 탭 "커뮤니티 탭과 완전히 동일" 명시) |

# 5. 공통 정책 요약

## 5-0. 수정·삭제 기간 운영 정책 (SSOT)

> 🔑 수정·삭제 기간 정책의 단일 소스(SSOT) — §1-2-A, §1-2-B, §1-3-A, §1-3-B 등 하위 정책은 본 절을 참조합니다.

> 🎛️ 정책의 핵심은 "N일"이 아니라 "운영 중 N일을 바꿀 수 있는 구조"입니다. 어드민 콘솔에서 코드 배포 없이 N일 설정을 변경할 수 있도록 설계하는 것이 의무이며, 구체 N값과 최초 기본값은 발주사가 운영 중 결정합니다.

### 설계 요구사항

| 항목 | 정책 |
| --- | --- |
| 설계 명제 | 어드민 콘솔에서 콘텐츠 유형별 수정·삭제 가능 기간(N일)을 운영 중 변경 가능한 파라미터로 구현. 코드 배포 불필요. |
| 설정 단위 | 3개 그룹 각각 독립 설정: ① 게시글 그룹(수정/삭제), ② 댓글 그룹(댓글·대댓글 수정/삭제), ③ 리뷰 그룹(수정/삭제) |
| 설정 가능 값 | 수정 불가 / N일 / 무제한 3가지 중 그룹별 선택 (어드민 §3.11에서 모드 선택). 참고 인코딩 — 0 = 수정 불가, N(자연수) = N일 이내, -1 = 무제한 |
| 변경 반영 시점 | 어드민 저장 즉시 반영 (실시간). 기존 콘텐츠도 변경된 값이 즉시 소급 적용 (예: 제한 7일 → 무제한으로 변경 시, 이미 7일 지난 글도 수정 가능해짐). |
| 적용 기준 | "작성 시각부터 N일 이내"인 콘텐츠만 수정·삭제 가능. 그룹 설정에 따라 분기 — 수정 불가(0): 작성 직후부터 수정·삭제 차단. N일(N≥1): 위 기준 적용. 무제한(-1): 작성 시각과 무관하게 항상 가능. |
| 안내 UX | N일 모드: N일 경과 후 ⋮ 메뉴에서 수정·삭제 항목 숨김 또는 비활성화. 시도 시 토스트: "작성 후 N일이 지나 수정·삭제할 수 없습니다." (문구 내 N은 현재 설정값). 수정 불가 모드: 작성 직후부터 항목 숨김, 시도 시 토스트 "이 콘텐츠는 수정·삭제할 수 없습니다." 무제한 모드: 항상 노출. |
| 최초 기본값 | 발주사 결정. 외주사는 배포 시점 기본값을 설정 가능하게만 설계(하드코딩 금지). |
| 삭제 시 DB 처리 | 사용자 삭제 클릭 시 화면에서는 즉시 숨김, 원본은 DB에 보관(soft delete). 게시글·댓글·리뷰 모두 동일 적용. 명예훼손 등 법적 이슈 대비. 구체 보관 기간은 추후 확정 (발주사 협의 필요). |

### 별도 적용 정책 (원칙 제외)

| 항목 | 정책 |
| --- | --- |
| 리뷰 삭제 의사 평점 재계산 | 리뷰 삭제 시 의사 평점 즉시 재계산 (삭제된 리뷰는 평균 산출에서 제외). DB 보관과 독립적으로 항상 적용. |
| 대댓글 있는 댓글 삭제 | N일 내에도 완전 삭제 안 됨 — §1-3-B "삭제된 댓글입니다" placeholder 대체 정책 적용. |

### 추가 주의 (인증 정책과의 상호작용)

- N일 경과로 수정 자체가 불가능한 콘텐츠는 자동으로 재인증도 불가. §3-4 "인증된 글 재인증"은 수정 권한이 있는 상태에서만 진입 가능.
- 어드민에서 N일을 줄이는 경우(예: 무제한 → 7일) 과거 인증된 글의 재인증 접근이 소급적으로 차단될 수 있으므로, 설정 변경 전 운영자에게 경고 메시지 표시 권장.

## 1) 공통 API/상태 처리 정책

### [공통 API 응답 상태]

- 모든 조회/저장 화면은 loading / success / empty / error 4상태를 정의합니다.
- loading: 스켈레톤 또는 로딩 인디케이터를 노출하고, 중복 요청을 막기 위해 주요 버튼을 비활성화합니다.
- empty: 빈 상태 아이콘 + 안내 문구 + 다음 행동 버튼(예: 다시 검색/작성하기)을 제공합니다.

## 2) 목록/정렬/페이지네이션 정책

### [페이지네이션]

- 목록형 화면(커뮤니티, 공지사항, 검색 결과)은 cursor 기반 페이지네이션을 사용합니다.
- 첫 요청은 cursor=null로 호출하고, 응답 nextCursor가 있을 때만 추가 로드를 수행합니다.
- 추가 로드 시 id 기준으로 중복 항목을 제거합니다.
- 더 이상 불러올 데이터가 없을 때 “모든 항목을 불러왔습니다.” 문구를 노출합니다.
### [정렬/필터]

- 정렬(최신순/인기순) 및 필터(질환/진료과/탭)는 서버 쿼리 파라미터로 전달합니다.
- 정렬/필터 변경 시 기존 목록과 cursor를 초기화하고 1페이지부터 재조회합니다.
- 화면 상단 총 건수는 현재 정렬/필터 결과의 서버 응답 기준으로 노출합니다.

- 세션 만료가 감지되면 자동으로 비로그인 상태로 전환하고 홈 화면으로 이동합니다.
- 비회원 기능 제한 배너는 로그인 완료 즉시 숨김 처리하고, 회원 전용 기능 버튼을 활성화합니다.
### [SNS 계정 통합]

- 동일 이메일(또는 동일 식별자) 계정 충돌 시 통합 안내 모달을 노출합니다.
- 통합 성공 시 닉네임/활동/저장 목록은 서버 병합 결과를 기준으로 즉시 갱신합니다.
- 통합 실패 시 고객지원 연결 경로를 제공합니다.

## 4) 커뮤니티/마이페이지 운영 정책 보강

### [신고 처리 상태 노출]

- 신고 접수 성공 시 토스트 “신고가 접수되었습니다.”를 노출합니다.
- 운영 상태에 따라 콘텐츠를 정상 / 검토중 / 숨김으로 구분해 노출합니다.
- 숨김 상태 콘텐츠는 본문을 블라인드 처리하고 상세 진입을 제한합니다.
### [동시성/정합성]

- 이미 삭제된 게시글/댓글/후기 진입 시 404 안내 후 목록으로 복귀합니다.
- 저장/삭제 요청 진행 중에는 버튼을 비활성화해 중복 탭을 방지합니다.
- 상세 화면에서 변경된 데이터는 목록 복귀 시 재검증(revalidate)합니다.

## 5) 검색 정책 보강 (통합검색/명의찾기)

### [검색어 처리]

- 입력값은 공백 정리, 소문자 정규화 후 매칭합니다.
- 단일 키워드는 동의어 사전 확장 매칭을 적용합니다.
- 다중 키워드는 기본적으로 AND 매칭 정책을 적용합니다.
### [검색 결과 노출]

- 통합검색은 섹션별 우선순위(명의 > 병원 > 커뮤니티)를 적용할 수 있습니다.
- 각 섹션은 최대 노출 개수를 지정하고, 초과 시 “더 보기”로 상세 화면 이동합니다.
- 결과가 없을 때는 추천 검색어 또는 인기 키워드를 함께 노출합니다.

## 6) 접근성/개인정보/분석 이벤트

### [접근성]

- 아이콘 버튼(닫기/정렬/삭제/공유 등)은 스크린리더 라벨을 제공합니다.
- 모달 오픈 시 포커스를 모달 내부 첫 요소로 이동하고, 닫을 때 이전 포커스로 복귀합니다.
- ESC 닫기 가능 모달은 일관된 키보드 동작을 보장합니다.
### [개인정보]

- 이메일/전화번호 등 민감 정보는 마스킹 정책을 적용해 노출합니다.
- 로그/분석 이벤트에는 원문 개인정보를 저장하지 않습니다.
- 탈퇴 후 보존/파기 대상 데이터는 정책 문서와 일치해야 한다.
### [분석 이벤트]

- 주요 이벤트(탭 진입, 버튼 클릭, 저장/삭제 성공·실패, 검색 실행, 모달 오픈)를 정의합니다.
- 이벤트에는 화면명, 액션명, 결과(성공/실패), 오류코드(해당 시)만 포함합니다.

> ⚠️ 삭제 공통 정책: 게시글·댓글·리뷰 모두 삭제 시 화면에서는 즉시 숨김 처리되지만, 원본은 DB에 보관(soft delete)됩니다. 명예훼손 등 법적 이슈 대비. 게시글·리뷰에 댓글이 있으면 "댓글 N개가 함께 삭제됩니다" 확인 팝업 후 연쇄로 함께 숨김 처리됩니다. 리뷰는 추가로 의사 평점이 즉시 재계산됩니다. 세부는 정책 문서 §5-0 (https://www.notion.so/36d426e29161817e9609c9c0e5a58a51) 참조.

## 문서 범위

본 문서는 UI/UX·정책·인터랙션을 정의합니다. 데모 화면의 게시글·숫자·댓글 등은 더미이며 실제 서비스 데이터와 동일하지 않습니다. API·신고 처리·저장소 등은 실서비스 설계에 맞게 별도 확정합니다.

## 개요

커뮤니티는 환자들이 병원·의사 방문 후기, 치료 경험, 궁금한 점을 공유하는 공간입니다.

### 핵심 가치

- 신뢰도: 병원 진료 인증 뱃지로 실제 경험 후기 구분
- 전문성: 의사 인증 뱃지로 의료진 조언 표시
- 안전성: 신고 기능으로 부적절한 콘텐츠 차단
---

## 1. 헤더

- 커뮤니티 페이지 상단 바: 좌측에 「커뮤니티」 제목만 (이 줄 안에는 돋보기 없음)
- 통합 검색: 반응형 웹 전역 레이아웃에서 홈·명의 찾기·커뮤니티일 때 메인 뷰 우상단 고정 돋보기(본문 스크롤과 무관). 커뮤니티 페이지 헤더 내부가 아님.

## 2. 카테고리 필터

- 표시 조건: 전역 검색 중이거나 특정 유저 필터 중일 때 자동 숨김 (AnimatePresence)
- 1차 필터 탭: searchFilter · userFilter 없을 때 표시. 검색/유저 필터 활성 시 1·2차 전체 숨김.
- 질병별 / 진료과별
- 2차 칩 (질병별): 전체, 인기 질환 칩, 「질환 검색」
- 2차 칩 (진료과별): 전체, 내과, 외과, 소아과, 피부과 등
### 질환 검색 버튼 동작

- 클릭 시 칩 행이 질환명 검색 입력창으로 교체 (슬라이드 애니메이션)
- 좌측 ← 버튼: 검색 모드 종료 → 칩 행 복귀
- 플레이스홀더: 질환명 검색...
- 실시간 필터링: DISEASE_LIST 전체에서 검색어 포함 항목만 표시
- 인기 질환 칩에 없는 질환 선택 시: 해당 질환명 칩이 X 버튼과 함께 표시
### 검색 결과 표시 규칙 (카테고리 필터 내부)

- 입력 전: 결과 칩 영역 미표시 (검색창만)
- 결과 있음: 부분 일치 질환명 최대 10개 칩 (카테고리 소스: 내부 질환 DB·질환 그룹 분류 기준)
- 결과 없음: 검색 결과가 없습니다. 다른 질환명을 검색해 보세요.
- 결과 선택: 질환 칩 탭 시 필터 적용 → 검색 모드 종료 → 검색어 초기화
- 필터 효과: 선택 질환과 정확히 일치하는 게시글만 목록에 남음, 총 건수 즉시 갱신
### 필터 결과 상태 정의 (피드 영역)

공통

- 결과 0개: 빈 상태 카드 (아이콘 + 아직 [선택 카테고리] 후기가 없어요 + 안내 문구)
- 결과 1개 이상: 게시글 카드 목록, 상단 총 n건 실시간 반영
- 결과가 많을 때: 페이지네이션 없이 같은 목록 영역에서 연속 표시, 스크롤로 탐색
- 정렬 유지: 최신순/인기순 상태를 필터 결과에 재적용
- 고정/스크롤 분리: 카테고리 필터, 정렬 헤더, 우상단 통합검색 진입점은 유지, 게시글 목록 영역만 스크롤
회원 (앱에서 회원 역할일 때 TSX에만 표시되는 항목)

- 결과 0개 안내: 첫 후기를 남겨보세요!
- 임시저장 배너(조건부), 우하단 글쓰기 버튼 노출
비회원 (게스트 역할일 때)

- 결과 0개 안내: 다른 카테고리를 선택해 보세요.
- 비회원 사용량 배너(조건부), 우하단 글쓰기 버튼 미노출
### 커뮤니티 내 검색 vs 통합 검색

- 카테고리 필터 내 「질환 검색」: 질환명으로 게시글 필터링 (커뮤니티 전용)
- 통합 검색 모달: 우상단 돋보기 → 명의/병원/커뮤니티 탭. 「커뮤니티에서 더 보기」는 커뮤니티 탭으로만 전환, 검색어는 커뮤니티 필터로 넘기지 않음

| ① 통신 중 잠금 | 상태를 바꾸는 버튼은 클릭 즉시 Disabled 또는 로딩 상태로 전환해 연타 차단. (단, 좋아요·저장·팔로우 등 토글 액션은 비활성화 대신 응답 전까지 같은 대상 재탭을 무시 — 낙관적 갱신) |
| ② 완료 즉시 화면 전환 | 게시글/댓글/후기 작성·삭제·신고·인증요청·탈퇴 등 결과 확정 액션은 성공 토스트와 함께 모달 닫기/다른 화면 이동으로 재제출 환경 자체를 제거. |
| ③ 재진입·재시도 처리 | 사용자가 의도적으로 같은 행동을 다시 시도할 때만 "이미 ○○되었습니다" 안내 노출. 통신 지연·자동 재시도로 시스템이 중복 차단한 경우엔 정상 결과 토스트만 노출. |
| ④ 서버 계약 (백엔드 협업 노티스) | 상태 변경 요청은 동일 요청이 두 번 도달해도 한 번만 반영되도록 서버에서 보장. UI 가드는 사용자 편의용이며, 최종 방어선은 서버. |

> 💡 클라이언트: API 호출 시 Debouncing/Throttling 처리 및 버튼 state 관리.

> 서버: Idempotency(멱등성) 보장 로직 설계. API 요청 헤더의 고유 키값 또는 DB Unique 제약조건을 통한 중복 Insert 방지.

> ⚠️ 본 문서의 모든 항목은 베타 AIGA 출시 범위에 포함됩니다.

# A. 개요 & 정책 기반

## 1. 개요

### 1-1. 문서 범위

본 문서는 의료진 인증 기능의 단일 진실 공급원(SSOT)입니다. 인증 버튼 노출 규칙, 인증 절차 모달, 인증 정책, 인증 상태 관리까지 인증 관련 전체 정의를 통합 관리합니다.

🟢 본 문서가 다루는 범위:

- 비회원/일반회원/의사 인증 회원 × 미인증/인증완료 의사 = 7개 시나리오에서 의료진 인증 버튼의 노출 여부와 상태
- 의료진 인증 버튼 클릭 → 본인인증 → 면허번호 입력 → 인증 완료 까지의 모달 플로우
- 면허번호 포맷 검증 / 재시도 한도 / 이력 저장 정책
- 인증 상태 관리 (미인증 → 인증완료 상태 전이)
- 작성자 표시 정책 (인증 시점 일괄 갱신)
🔴 본 문서가 다루지 않는 범위 (위임):

| 주제 | 위임 문서 |
| --- | --- |
| 의사 프로필 화면 전체 구조 / 로그인 유도 모달 / 리뷰 작성 모달 | 공통 UI 컴포넌트 정의서 |
| 프로필 편집 모달(ProfileEditor) / 의사 정보 수정 요청 모달(DoctorInfoRequestModal) — 구성 · 필드 · 동작 | 데모 화면 SB (의사인증_v0.5 프로토타입 · ScreenSpecification.tsx "프로필 편집 모달 구성" 섹션) |
| 리뷰 작성 권한 / 리뷰 유형 분류 (환자/전문가) / 작성자 표시 UI 렌더링 | AIGA 사용자 등급별 정책 화면정의서 |

의사 프로필 화면에 도달하는 4가지 경로는 모두 동일하며, 도달 후 관찰자 유형과 대상 의사 상태에 따라 인증 버튼 노출이 분기됩니다 (§3 참조).

| 진입 경로 | 설명 |
| --- | --- |

| 홈 탭 → 인기 명의 섹션 → 의사 카드 클릭 | 홈 탭 내 주요 질환 인기 명의 카드 선택 |

## 2. 인증 등급 및 주체 분리 원칙

### 2-1. 인증 등급 (3-Tier 사용자 모델)

| 등급 | 식별 | 주요 권한 |
| --- | --- | --- |
| 비회원 | 로그인 없이 접근 | 열람 전용 |
| 일반 회원 | 로그인 + 이메일/SNS 가입 | 리뷰 작성, 정보 수정 요청, 인증 요청 |
| 의사 인증 회원 | 일반 회원 + 면허번호 인증 완료 | 본인 프로필 직접 편집, 전문가 리뷰 작성 |

### 2-2. 인증 주체 분리 원칙

AIGA의 의사 정보는 두 개의 독립적인 레이어로 관리됩니다.

① 의사 프로필 (Doctor Profile)

- AIGA 서비스에 등록된 의사 정보 DB (kormedi.com (http://kormedi.com/) 기반 수집)
- 해당 의사의 AIGA 가입 여부와 무관하게 존재
- 모든 프로필은 기본적으로 "미인증" 상태로 시작
② AIGA 계정의 인증 상태

- 모든 사용자는 동일한 방식으로 가입 (계정 타입 구분 없음)
- 가입 후 면허번호 인증 여부에 따라 상태가 달라짐:
- 일반 사용자 → "일반 회원" 상태 유지
- 의사 → 면허번호 인증 후 "의사 인증 회원" 상태 획득
### 2-3. 런칭 버전에서 발생 가능한 경우

| 경우 | 의사 프로필 | AIGA 계정 | 설명 |
| --- | --- | --- | --- |
| A | 있음 | 없음 | 의사 정보는 DB에 있으나 해당 의사는 AIGA 미가입 |
| B | 있음 | 있음 (일반) | 의사가 AIGA 가입했으나 아직 면허 인증 미완료 |
| C | 있음 | 있음 (인증 완료) | 의사가 가입 + 인증 완료 → 본인 프로필 직접 편집 가능 |

런칭 버전 제약사항:

- 면허번호 포맷(자릿수·형식)이 유효하면 자동 승인됩니다.
- 입력된 면허번호는 해당 계정과 연결되어 저장됩니다.
핵심 원칙:

프로필과 계정은 "의사 인증" 절차를 통해서만 연결됩니다. 인증 없이는 누구도 특정 의사 프로필을 직접 편집할 수 없습니다.

## 3. 인증 버튼 노출 규칙 (시나리오별)

의료진 인증 버튼은 의사 프로필 화면 본문의 액션 버튼 영역에 노출됩니다. 관찰자 유형과 대상 의사 인증 상태에 따라 노출 여부 및 상태가 결정됩니다.

### 3-1. 시나리오별 통합 매트릭스

| # | 관찰자 | 대상 의사 | 인증 버튼 상태 | 버튼 클릭 시 동작 |
| --- | --- | --- | --- | --- |
| 1 | 일반회원 | 미인증 | ✅ active | VerificationFlow 모달 오픈 (§5) |
| 2 | 일반회원 | 인증 완료 | ❌ 미노출 | — |
| 3 | 의사 인증 회원 | 미인증 (타인) | ❌ 미노출 | — |
| 4 | 의사 인증 회원 | 인증 완료 (타인) | ❌ 미노출 | — |
| 5 | 의사 인증 회원 | 본인 (인증완료) | ❌ 미노출 | — (대신 "내 프로필 수정" 등 본인 관리 진입점이 노출됨 — 데모 SB doctor-self 시나리오 참조) |
| 6 | 비회원 | 미인증 | ✅ disabled | 로그인 유도 모달 오픈 (※ 공통 UI 정의서 위임) |
| 7 | 비회원 | 인증 완료 | ❌ 미노출 | — |

### 3-2. 노출 규칙 요약

- active 조건 — 일반회원이면서 대상 의사가 미인증 상태일 때 (시나리오 1)
- disabled 조건 — 비회원이면서 대상 의사가 미인증 상태일 때 (시나리오 6, 로그인 유도 목적)
- 미노출 조건 — 그 외 모든 경우
- 대상 의사가 이미 인증 완료된 경우: 본인 인증이 무의미하므로 미노출
- 의사 인증 회원이 타인 미인증 의사를 보는 경우: 본인이 아닌 타인의 인증을 대신 수행할 수 없으므로 미노출
- 의사 인증 회원이 본인 프로필을 보는 경우: 이미 인증 완료 상태이므로 인증 버튼 미노출. 단, 본인 프로필 관리를 위한 별도 진입점("내 프로필 수정")이 노출되며, 이는 프로필 편집 모달(ProfileEditor)을 엽니다 (세부 사양은 데모 SB doctor-self 시나리오 참조)
### 3-3. 비회원 진입 시 UX 정책

기본 정책:

- 액션 버튼은 물리적으로 노출하되 비활성화 (disabled) 상태 유지
- 비회원이 비활성 버튼 클릭 시 로그인 유도 모달 노출
- 로그인 유도 모달은 [공통 UI 컴포넌트 정의서] 참조
설계 의도:

- 비회원에게 "회원가입 시 열리는 기능"을 미리 보여줘 전환 유인 확보
- 클릭 시 기능을 즉시 차단하지 않고 회원 전환 유도

# B. 인증 절차

## 4. 인증 절차 개요

의료진 인증은 2단계 모달로 진행됩니다.

[의료진 인증 요청 버튼 클릭]
       ↓
[Step 1. 본인인증]
   PASS / 카카오 / 네이버 중 선택 + 본인확인 약관 동의
       ↓
[Step 2. 면허번호 입력]
   의료 면허번호 입력 + 의사 자격 검증 약관 동의
       ↓
[인증 완료]
   계정 표시명 갱신 (닉네임 → 실명)
   의사 인증 뱃지 부여
   기존 작성물 작성자 표시 일괄 갱신 (§8 참조)

설계 의도:

- Step 1 본인인증으로 "이 면허번호의 주인 = 지금 인증하는 계정"임을 강제 → 신원 도용 리스크 1차 차단
- Step 2 면허번호 입력으로 의사 자격 확인

## 5. VerificationFlow 모달

### 5-1. 모달 기본 정보

| 항목 | 내용 |
| --- | --- |
| 진입 조건 | 의료진 인증 요청 버튼 클릭 시 (active 상태일 때만) |
| 진입 위치 | 미인증 의사 프로필 화면 본문 액션 버튼 영역 |
| 닫기 방법 | 우상단 X 버튼, 취소 버튼 |
| 단계 수 | 2단계 (Step 1: 본인인증 → Step 2: 면허번호 입력) |
| 이탈 경로 | 닫기(X) → 모달 닫힘, 인증 상태 변화 없음 / 인증 성공 → 모달 닫힘 + 프로필 화면 갱신 / 인증 실패 → 재시도 또는 닫기 |

    end

# C. 인증 정책

## 6. 면허번호 검증 정책

### 6-1. 현재 결정 사항 (런칭 베이스)

| 항목 | 정책 |
| --- | --- |
| 검증 단계 수 | 2단계 (본인인증 → 면허번호 입력) |
| 필요 정보 | 의료 면허번호 (입력 포맷: 숫자만, 4자리 이상) |
| 검증 방식 | 면허번호 포맷(자릿수·형식) 검증 |
| 승인 기준 | 숫자 4자리 이상 포맷이 유효하면 자동 승인 |
| 재시도 한도 | 일일 5회 (계정 단위 + IP 단위 둘 다 카운트) |
| 리셋 주기 | 매일 자정(KST) 기준 리셋 |

> ※ 면허번호 검증: 입력된 면허번호의 포맷(숫자 4자리 이상)만 확인해 자동 승인합니다.

### 6-2. 검증 프로세스

면허번호 검증은 포맷 검증으로 진행됩니다:

- 포맷 검증: 입력값이 "숫자 4자리 이상"인지 확인. 포맷이 유효하면 자동 승인, 너무 짧거나 숫자가 아닌 글자가 섞이면 포맷 오류로 처리
프로세스 (런칭 베이스):

[본인인증 통과] → [입력] → [포맷 검증: 숫자 4자리 이상]
  → [재시도 한도 확인] → [포맷 유효 시 자동 승인]

- 포맷(숫자 4자리 이상)이 유효하면 즉시 승인
### 6-3. 미승인 사유 분류

베타 단계에서는 포맷이 유효하면 자동 승인되므로, 미승인 사유는 포맷 오류·중복 인증·한도 초과로 한정됩니다.

베타 미승인 사유 (3가지):

| 사유 | 판별 가능 여부 | 메시지 공개 수위 |
| --- | --- | --- |
| 포맷 오류 | ✅ (숫자 4자리 이상 여부로 판별) | 공개 |
| 중복 인증 | ✅ (AIGA 내부 판별 가능) | 숨김 (안전형) — "해당 의사가 이미 AIGA 회원"임을 노출하지 않음 |
| 재시도 한도 초과 | ✅ (계정·IP 카운트 일일 5회 도달 여부로 판별, §6-4) | 공개 (한도 및 리셋 시점 안내) |

> 오류 메시지 본문 4종은 §5-6 참조.

### 6-4. 재시도 제한 정책

한도 규칙:

- 일일 5회 (계정 단위 + IP 단위 둘 다 카운트)
- 계정 카운트: 로그인된 사용자 ID 기준
- IP 카운트: 요청이 들어온 IP 주소 기준
- 두 카운트 중 어느 하나라도 5회 도달 시 인증 차단 (한 IP에서 여러 계정으로 시도하는 우회 시도 방지)
- 리셋 주기: 매일 자정(KST) 기준 리셋
한도 도달 시 동작:

- 입력 화면은 정상 노출
- "면허번호 확인" 버튼만 비활성화 + 안내 메시지 노출
- 입력 필드 자체는 노출 유지 (과도한 차단 방지)
> ※ 초기 정책: 5회/일 한도와 자정 리셋은 사용자 불편 최소화를 위한 기본값. 어뷰징 사례 발생 시 한도 축소(예: 3회) 또는 잠금 기간 연장 가능.

### 6-5. 미승인 시도 이력 저장 정책

기본 원칙: 평상시 시도는 저장하지 않음, 한도 초과 시에만 저장

저장하지 않는 경우:

- 일일 5회 한도 내 일반 미승인 시도
- 프라이버시 존중 및 저장 공간 효율 목적
저장하는 경우 (한도 초과 시):

| 저장 항목 | 저장 방식 | 용도 |
| --- | --- | --- |
| 계정 ID | 평문 | 동일 계정의 반복적 우회 패턴 추적 |
| IP 주소 | 평문 | 동일 IP에서 여러 계정 사용 패턴 추적 |
| 시도 일자/시간 | 평문 | 어뷰징 타임라인 분석 |
| 시도 횟수 | 평문 | 악의 정도 판별 |
| 입력된 면허번호 | SHA-256 해시 | 동일 번호 반복 시도 패턴만 비교 (원본 복원 불가) |

저장 기간: 90일 보관 후 자동 삭제 (개인정보 최소 수집 원칙 준수)

저장 이력 활용:

- 운영자 모니터링 대시보드에서 어뷰징 의심 계정/IP 조회
- 반복 위반 시 계정 잠금 또는 IP 차단 검토
- 법적 대응 시 증거 자료로 활용 가능
> ※ 초기 정책: 해시 방식 및 90일 보관은 개인정보 보호를 우선한 기본값. 법무 자문 후 보관 기간 조정 가능 (최대 1년 이내 권장).

## 핵심 원칙 두 가지

의사 인증이 완료되면 다음 두 가지가 일관되게 적용됩니다.

1. 이름이 닉네임에서 실명으로 바뀝니다 (예: 「건강맘」 → 「구자호 교수」)
1. 이름 옆에 「의사 인증」 뱃지가 붙습니다
닉네임은 인증 후에는 변경할 수 없으며, 화면에서 닉네임 입력란 자체가 사라집니다.

## 🎨 인증 의사 콘텐츠 시각 강조 — 디자인 토큰 (전체 탭 공통)

인증 의사가 작성한 모든 콘텐츠(게시글·댓글·대댓글·리뷰 답글)는 단일 디자인 토큰을 공유합니다. 모든 탭(홈·커뮤니티·리뷰 등)에서 동일한 색상으로 일관되게 적용하세요.

| 요소 | 토큰명(권장) | 라이트 모드 | 다크 모드 | 적용 위치 |
| --- | --- | --- | --- | --- |
| 의사 콘텐츠 배경 | --surface-doctor | #EFF6FF (옅은 파란빛) | rgba(59,91,219,0.15) | 게시글 카드, 댓글/대댓글 말풍선, 리뷰 답글 박스 |
| 의사 인증 뱃지 | --badge-doctor | 배경 #E1F5EE / 텍스트 #0F6E56 / 도트 #1D9E75 | (라이트와 동일 또는 디자인팀 결정) | 이름 옆 뱃지 |
| 의사 아바타 | --avatar-doctor | 배경 #DBEAFE / 텍스트 #1E40AF | 배경 #1E3A8A / 텍스트 #BFDBFE | 실명 이니셜 아바타 |
| 일반 댓글 들여쓰기 | --border-comment-indent | 2px solid #E5E7EB | 2px solid #374151 | 대댓글 좌측 보더 |

> 📌 표현 통일 규칙: 본문에서 "옅은 파란빛", "옅은 블루-그레이" 등 다양하게 표현되더라도, 실제 구현은 모두 --surface-doctor 토큰 하나를 공유합니다. 콘텐츠 유형별로 다른 색을 쓰지 않습니다.

> 🔍 이유: 사용자가 "의사 인증 회원의 콘텐츠"라는 한 가지 의미를 한 가지 색으로 인지하면 학습 비용이 낮아지고, 디자인 시스템 측면에서도 토큰 1개로 일관 관리할 수 있습니다.

## 1. 홈 탭

### 인증 의사 본인이 보는 홈

홈 탭 자체의 구성은 일반 회원과 동일합니다. 다만, 내가 쓴 글이 「커뮤니티 추천글」 영역에 노출될 경우, 다른 사용자에게는 내 이름이 실명 + 뱃지로 표시됩니다.

### 다른 사용자가 보는 홈

「커뮤니티 추천글」 카드에서 작성자가 인증 의사인 경우 다음이 보입니다.

- 작성자 이름: 실명 + 「의사 인증」 뱃지
- 카드 배경: --surface-doctor 토큰 (옅은 파란빛 — 전체 탭 공통)
> 🔍 사용자 경험 포인트: 홈을 처음 열었을 때 "이 앱에는 진짜 의사가 활동한다"는 인상을 자연스럽게 전달합니다. 의사 글만 따로 강조하지 않고 일반 글과 섞여서 보이되, 한눈에 구분되는 것이 목표입니다.

## 2. AI 챗봇 탭 (AIGA)

### 인증 의사 본인이 보는 챗봇

챗봇 화면은 일반 회원과 동일합니다. 별도 권한이나 다른 화면이 없습니다.

### AI가 의사를 추천할 때

AI가 답변에 의사를 추천하면 카드 형태로 보여주는데, 그 의사가 AIGA에서 인증된 회원이라면 카드 위에 「의사 인증」 뱃지가 함께 표시됩니다.

> 🔍 사용자 경험 포인트: AI가 추천하는 의사 카드는 보통 사진과 이름만 짧게 보입니다. 여기에 뱃지 하나가 붙는 것만으로 "이 의사는 단순 데이터가 아니라 실제로 우리 앱을 쓰는 의사구나"라는 신뢰가 생깁니다.

### UX 시나리오: 챗봇 추천 → 의사 프로필 이동

| 단계 | 사용자가 하는 행동 | 화면에서 보이는 것 |
| --- | --- | --- |
| 1 | 증상을 챗봇에 입력 | AI가 답변하며 추천 의사 카드 노출 |
| 2 | 카드의 의사가 인증 회원인 경우 | 카드 프로필 우측에 「의사 인증」 뱃지 |
| 3 | 카드 탭 | 의사 프로필 모달로 이동 |

## 3. 명의 찾기 탭

### 검색 결과 목록에서

의사 이름을 검색하면 결과 카드에 다음이 보입니다.

- AIGA 인증 의사 → 이름 옆에 「의사 인증」 뱃지
- 같은 이름의 의사가 여러 명일 경우 → AIGA에서 직접 인증한 의사가 목록 맨 위에 노출
- 같은 이름 케이스에는 「정확한 의사 프로필」 같은 작은 안내 문구를 붙여 사용자가 헷갈리지 않게 함
- 인증 의사 본인이 검색했을 때: 본인 프로필 카드에 「본인」 표시가 추가로 붙음 (다른 동명이인과 구분 명확화)
### 본인 프로필 카드의 인터랙션 차이

인증 의사가 검색 결과나 추천 카드에서 본인 프로필을 만났을 때:

- 즐겨찾기(저장) 버튼: 숨김 처리 (자기 자신을 즐겨찾기하는 무의미한 동작 차단)
- 리뷰 작성 버튼: 숨김 처리 (본인 리뷰 작성 불가 — 정책 SSOT: 📋 사용자 등급별 정책 화면정의서 (https://www.notion.so/36d426e29161817e9609c9c0e5a58a51) §2-1)
- 「내 프로필 수정」 버튼: 추가 노출 (마이페이지의 같은 진입점과 동일하게 동작)
### 의사 프로필 상세 화면을 열었을 때

- 다른 사용자가 보는 경우: 평소와 동일한 프로필 화면 — 상단 뱃지는 「의사 인증」 뱃지로 표시 (게시물 등과 동일)
- 인증 의사 본인이 자기 프로필을 연 경우: 화면 상단에 「내 프로필 수정」 버튼이 추가로 보임. 이때 상단 뱃지는 「의사 인증」이 아니라 「인증 완료」 뱃지로 표시됨 (본인 프로필 모달 전용)
- 이 버튼을 누르면 마이페이지의 프로필 수정 화면으로 이동
- 수정 가능한 항목: 진료 분야 키워드, 자기소개, 진료 시간, 논문/경력 등
- 수정 불가 항목: 실명, 면허번호, 소속 병원
> 🔍 사용자 경험 포인트: 의사 입장에서 "내 프로필을 내가 직접 관리할 수 있다"는 체감은 인증의 가장 큰 보상입니다. 명의 찾기 화면에서 바로 수정 진입점을 두는 것이 핵심입니다.

### UX 시나리오 1: 환자가 검색해서 의사를 찾는 흐름

## 4. 커뮤니티 탭

의사 인증의 효과가 가장 두드러지는 탭입니다.

### 글 목록에서

인증 의사가 쓴 게시글 카드에는 다음이 표시됩니다.

- 작성자 이름: 실명 + 「의사 인증」 뱃지
- 카드 배경: --surface-doctor 토큰 (옅은 파란빛 — 전체 탭 공통)
### 글 상세 화면(댓글/대댓글)에서

인증 의사가 단 댓글은 다음과 같이 보입니다.

- 댓글 말풍선 배경: --surface-doctor 토큰 (옅은 파란빛 — 게시글 카드와 동일 색상)
- 이름 옆: 「의사 인증」 뱃지
- 대댓글도 동일한 규칙 적용
> 🔍 사용자 경험 포인트: 환자 입장에서 "이 댓글이 진짜 전문의에게서 온 것"임을 한눈에 알 수 있어 플랫폼 신뢰도가 크게 올라갑니다. 동시에 일반 회원의 댓글이 묻히지 않도록 강한 강조(예: 상단 고정)는 피하고, 배경 톤 차이로만 부드럽게 구분합니다.

### 작성자 프로필 모달 (목록에서 작성자 탭 시)

커뮤니티 글 목록에서 인증 의사의 작성자 이름·아바타를 탭하면 작성자 프로필 모달이 뜹니다.

- 진입 위치: 글 목록에서만 (작성자 이름/아바타 탭)
- 모달 구성: 아바타 + 실명 + 「의사 인증」 뱃지 + "작성한 글 N개"
- 버튼 2종
- 주 버튼 「작성한 글 보기」: 해당 회원이 커뮤니티에 쓴 글 목록으로 이동
- 보조 버튼 「의사 프로필 보기」: 의사 프로필 모달(DoctorProfileModal)로 이동 — 홈·AI 챗봇·명의 찾기와 공통으로 쓰는 모달
- 모달 전환 규칙: 모달 위에 모달을 쌓지 않고 교체(replace) — 「의사 프로필 보기」를 누르면 작성자 프로필 모달은 닫히면서 의사 프로필 모달로 전환됨
> ⚠️ 상세 화면에서는 비활성: 게시글 상세 모달 안에서는 작성자(또는 댓글 작성자)의 이름·아이콘을 터치해도 반응이 없습니다. 작성자 프로필 모달은 목록 화면에서만 진입합니다.

> 🔍 사용자 경험 포인트: 목록에서 "이 사람 누구지?"라는 호기심이 생기는 지점에만 가볍게 프로필을 띄우고, 상세 열람 중에는 흐름이 끊기지 않도록 합니다. 더 깊은 정보(의사 프로필)는 보조 버튼으로 한 단계 뒤에 둡니다.

### 상단 프로필 영역의 차이

| 항목 | 일반 회원 | 인증 의사 |
| --- | --- | --- |
| 아바타(아이콘) | 기본 사람 아이콘 + 그라데이션 | 실명 이니셜 아바타 + 파란 그라데이션 |
| 화면 헤더 | 「마이페이지」 | 「마이페이지」 (뱃지 없음) |
| 이름 표시 | 닉네임 | 실명 (변경 불가) |
| 이메일 표시 | 이메일 주소 | 이메일 주소 + 「의사 인증」 뱃지 |

> 📌 뱃지 표기 규칙 (화면 성격별): 마이페이지 헤더 뱃지는 「의사 인증」으로 유지합니다. 본인이 자기 상태를 보는 화면이지만, 바로 아래 인증 정보 카드의 인증 상태 = 「인증 완료」가 상태를 이미 명시하므로, 헤더는 다른 화면과 동일한 신뢰 라벨(「의사 인증」)로 둡니다. ※ 명의 찾기 의사 프로필 상세 모달만 본인 뷰 한정으로 헤더 뱃지를 「인증 완료」로 표기합니다(화면 성격이 '내 명함 점검'=상태 알림에 가깝기 때문).

### 닉네임 카드 → 인증 정보 카드로 교체

기존의 닉네임 입력 카드(필수, 2~10자, 저장 버튼)가 통째로 사라지고, 그 자리에 인증 정보 카드가 들어옵니다.

- 실명: 자물쇠 아이콘 + 「인증 정보」 라벨, 읽기 전용
- 소속 병원: 읽기 전용 (예: 서울대학교병원)
- 전문 분야: 읽기 전용 (예: 정형외과 · 척추외과)
- 인증 상태: 「인증 완료」 표시 (뱃지 톤)
- 하단 버튼: 「의사 프로필 수정」 → 탭 시 의사 프로필 편집 화면으로 이동
편집 가능한 항목과 불가능한 항목이 명확하게 구분되어야 합니다.

- ✅ 편집 가능: 진료 분야 키워드, 자기소개, 논문/경력, 진료 시간 등
- 🔒 편집 불가: 실명, 면허번호, 소속 병원
### 내 활동 영역

기존 「게시글 / 댓글 / 후기 / 저장」 4탭 구조는 유지되며, 다음이 달라집니다.

- 내가 쓴 모든 글/댓글/후기 카드의 작성자 표기가 실명 + 뱃지로 자동 갱신됨
- 인증 이전에 닉네임으로 썼던 글들도 동일하게 적용됩니다 (이 부분은 인증 시작 전에 사용자에게 명확히 안내해야 함)
> 🔗 관련 정책 문서와 일치: 🩺 의사 인증 SB(화면/정책/기능정의서) v1.0(외주) (https://www.notion.so/36d426e2916181658e05eb431f8d21d3) §8 작성자 표시 정책에 동일한 정책이 명시되어 있습니다 — “인증 승인 시점으로 모든 작성물을 실명+뱃지로 일괄 갱신 / 신뢰성 일관성, 혼란 방지”. 두 문서는 동일한 정책을 공유합니다.

> 예시: 김민수 교수 본인의 마이페이지 「내 활동 → 후기」 탭

> - 인증 완료 시점에 과거 작성물 일괄 갱신 배치 처리

> 아키텍처 구현 요구사항:

> - author_id 기준 join 방식 권장: 인증 승격 즉시 자동 갱신이 이루어짐

> - authorName을 복사 저장하는 구조라면 인증 승격 시점에 배치(cron) 일괄 갱신 필요

> - 인증 시작 전 동의 화면 필수: "인증 완료 시 과거에 닉네임으로 쓴 모든 글/댓글/후기가 실명·뱃지로 일괄 전환됩니다" 명시 + 체크박스 동의

### 후기 탭의 특수 케이스

인증 의사 본인이 명의 찾기에서 자기 자신의 프로필을 「저장」하거나 후기를 남기는 행위를 차단(댓글은 가능)

### 회원 탈퇴 시

- 인증 의사가 탈퇴하면 작성한 글/댓글/리뷰의 작성자 표기를 어떻게 보존할지 발주사 확정 필요 (옵션 후보: 「탈퇴한 회원」 vs 「인증 의사(탈퇴)」). 콘텐츠 자체는 §5-0 공통 정책 적용 (화면 숨김 + DB soft delete).
- 면허번호, 소속 등 민감한 인증 데이터의 보관 및 파기 기간은 개인정보처리방침과 함께 명시 필요 (발주사 주도 결정).
### 「저장」 탭 (인증 의사 케이스)

일반 회원과 완전 동일하게 동작합니다. 「저장」 탭은 본인이 즐겨찾기한 의료진 목록이고, 인증 여부와 무관하게 동일한 인터랙션을 제공합니다.

- 본인 프로필에서는 「본인을 저장」하는 즐겨찾기 버튼이 숨김 처리됨 (명의 찾기 탭에서도 동일)
> 🔍 사용자 경험 포인트: 마이페이지만 봐도 "나는 AIGA에서 인증된 의사다"라는 자기 인식이 명확해집니다. 닉네임 입력란이 사라지고 자물쇠 아이콘이 붙는 변화가 의사에게는 일종의 신뢰 보상으로 작용합니다.

### UX 시나리오: 인증 직후 첫 마이페이지 진입

| 단계 | 사용자가 하는 행동 | 화면에서 보이는 것 |
| --- | --- | --- |
| 1 | 인증 완료 직후 마이페이지 진입 | 이메일 옆에 「의사 인증」 뱃지가 새로 붙어 있음 (헤더는 「마이페이지」만) |
| 2 | 아바타 영역 확인 | 사람 아이콘이 실명 이니셜로 변경됨 |
| 3 | 프로필 영역 스크롤 | 닉네임 카드 사라지고, 인증 정보 카드(실명 · 소속 · 전문 · 인증 상태) 등장 |
| 4 | 「의사 프로필 수정」 탭 | 진료 분야 키워드, 자기소개 등 편집 화면으로 이동 |
| 5 | 실명 항목 탭 시도 | 자물쇠 아이콘과 함께 "인증 정보는 변경할 수 없습니다." 안내 |

## 6. 리뷰 탭 — 리뷰 작성 강화 · 댓글/대댓글 추가

이번 베타에서 리뷰 탭은 두 가지가 강화됩니다 — 6-1) 리뷰 본문 작성에 이미지 첨부·병원 진료 인증 추가, 6-2) 리뷰 댓글/대댓글 기능 추가(댓글/대댓글 작성, 좋아요(공감), 신고하기 포함).

의도: 의사 인증 회원이 리뷰에 댓글로 직접 답할 수 있게 해 환자–의사 소통을 제공하고, 기존 텍스트 전용 리뷰를 커뮤니티 탭과 동일한 작성·인터랙션 경험으로 통일하기 위함입니다. 그 결과 기존 텍스트 위주이던 리뷰에 이미지·인증·댓글·좋아요(공감)·신고 등 여러 기능이 추가됩니다. 따라서 두 기능 모두 커뮤니티 탭과 동일한 정책을 따릅니다.

### 6-1. 리뷰 본문 작성 — 이미지 첨부 · 병원 진료 인증

> 🆕 베타 추가: 기존 리뷰는 텍스트만 작성 가능했으나, 이번 베타부터 리뷰 본문 작성 시 이미지 첨부 + 병원 진료 인증이 추가됩니다. 정책은 커뮤니티 글쓰기와 동일합니다.

> ⚠️ 아래 「핵심 기능 정의」의 댓글/대댓글 = 텍스트만 규칙은 리뷰에 달리는 댓글에 한정되며, 리뷰 본문 작성(이 절)과는 별개입니다.

- 이미지 첨부: 최대 10장, 장당 5MB, JPG·PNG·WEBP — 커뮤니티 글쓰기와 동일
- 병원 진료 인증 (선택): 영수증·처방전 업로드 시 OCR 자동 매칭으로 신뢰 뱃지 부여. 인증하지 않아도 리뷰 작성 가능. 커뮤니티 글쓰기의 「🏥 병원 진료 인증하고 신뢰 배지 받기」와 동일 정책(선택)
- 임시저장: 텍스트·별점만 복원, 이미지·인증 파일은 임시저장 제외 (공통 UI 컴포넌트 정의서 §1-3과 동일)
- 적용 대상: 리뷰를 작성하는 모든 회원(일반 회원·인증 의사 공통). 의사 인증 회원 전용 기능이 아니며, 리뷰 작성 팝업 공통 사양은 🧩 공통 UI 컴포넌트 정의서 (https://app.notion.com/p/36d426e291618194a17ad5193a3deac6) §1-3 참조
> 📌 표현 정합성 메모(개발 참고): 데모 코드의 리뷰 인증 영역 안내문구가 "인증하면 리뷰 작성이 가능합니다"로 필수처럼 읽히나, 라벨은 「(선택)」이고 본 정책도 선택입니다. 커뮤니티와 동일하게 "선택" 톤으로 통일하세요. 또한 인증 처리 방식은 영수증·처방전 OCR 자동 매칭(커뮤니티와 동일)이며, 데모의 「인증 요청 완료 / 승인되면」 안내가 수동 검토처럼 읽힐 수 있으니 OCR 자동 매칭 톤으로 정리하세요.

> 🆕 베타 추가: 리뷰 아래에 댓글/대댓글 기능이 새로 추가됩니다. 일반 회원·인증 의사 모두 작성할 수 있으며, 인터랙션 방식은 커뮤니티 탭의 게시글 댓글/대댓글과 동일합니다.

### 시각 강조 — 인증 의사 콘텐츠

인증 의사가 단 댓글·대댓글은 시각적으로 구분됩니다. 이 정책은 홈·커뮤니티·리뷰 탭에서 일관되게 적용됩니다.

- 말풍선 배경: 옅은 파란빛 (#EFF6FF, 다크모드 rgba(59,91,219,0.15))
- 작성자 이름 옆: 「의사 인증」 뱃지
- 아바타: 실명 이니셜 + 파란 톤 (#DBEAFE / #1E40AF)
- 작성 시간: "방금 전", "2시간 전" 등 상대 시간
### 일반 회원이 보는 리뷰 탭

- 환자 리뷰 목록 열람
- 리뷰 아래 댓글/대댓글 작성 가능
- 작성한 댓글은 일반 회색 말풍선으로 표시
- 본인이 쓴 리뷰인 경우: "본인 프로필에는 리뷰를 작성할 수 없습니다" 안내 (단, 댓글/대댓글은 작성 가능)
### 인증 의사가 보는 리뷰 탭

- 환자 리뷰 목록 열람 + 댓글/대댓글 작성
- 작성한 댓글은 옅은 파란 말풍선 + 인증 뱃지
- 본인 프로필의 리뷰에도 댓글/대댓글로 직접 답변 가능

## 3\.2 신고 처리

### 화면 목적

### 주요 컴포넌트

- 탭: 대기중 / 완료
- 사유 필터: 8종 셀렉트 \(전체 \+ 광고/홍보성·욕설/인신공격·허위 정보·개인정보 노출·중복 게시글·도배/스팸·의료법 위반 소지·기타\) — data\-reason 기반 작동\. 선택 시 즉시 해당 사유의 행만 표시, 결과 건수 토스트 알림
- 컬럼: 상태 / 유형 / 내용 / 사유 / 신고 시각 / 처리
- 유형 컬럼 표시: 게시글 / 후기 / 게시글 댓글 / 후기 댓글 — 댓글은 부모 콘텐츠를 라벨에 직접 표시하여 4종으로 명시

### 행 펼침/접힘 \(UI 상세\)

- 행 좌측의 ▶ 아이콘 또는 행 전체를 클릭하면 펼침/접힘 토글\. 펼침 상태에서는 아이콘이 ▼ 로 변경

	- • 콘텐츠 본문 전문 \(게시글·후기는 전체 텍스트, 댓글은 부모 게시글·후기 본문도 함께 표시\)
	- • 작성자 정보: 닉네임\(마스킹\), 가입일, 신고 누적 건수 \(3건 이상 시 빨간색\)
	- • 신고자 정보: 닉네임\(마스킹\), 신고 시각, 신고 사유, 신고자가 작성한 자유 텍스트 메모
	- • 병원 진료 인증 여부: 후기 신고의 경우 영수증·처방전·예약확인서 인증 통과 여부 표시 \(인증된 후기에 빨간색 "인증됨" 배지\)
	- • 첨부 이미지: 후기에 사진이 있으면 펼쳐서 함께 표시

## 3\.3 커뮤니티 관리

### 화면 목적

커뮤니티 게시글과 댓글을 조회하고, 예외 케이스에 한해 선제적으로 삭제·복원합니다\.

### 주요 컴포넌트

- 탭: 전체 / 게시글 / 댓글 \(게시글\-댓글 트리 구조\)
- 검색: 작성자·키워드
- 컬럼: 유형 / 내용 / 작성자 / 작성일 / 상태 / 처리
- 댓글 행은 게시글 행 바로 아래에 들여쓰기 \+ ↳ 아이콘으로 부모 관계 시각화

### 행 펼침/접힘 \(UI 상세\)

- 행 좌측의 ▶ 아이콘 또는 행 전체를 클릭하면 펼침/접힘 토글\. 펼침 상태에서는 아이콘이 ▼ 로 변경
- 펼침 영역 구성:
	- • 게시글: 본문 전문 \+ 작성자 닉네임\(마스킹\)·가입일 \+ 좋아요·댓글·조회수 카운터 \+ 작성 시 입력한 질환 태그·해시태그
	- • 댓글: 부모 게시글 본문 요약\(클릭 시 부모로 이동\) \+ 댓글 본문 전문 \+ 작성자 정보
	- • 작성자의 신고 누적 건수: 0건은 회색, 1~2건은 주황, 3건\+ 빨간색 표시
- 동일 시점에 여러 행 펼침 가능

### 병원 진료 인증 배지 \(UI 상세\)

- 게시글 작성 시 영수증·처방전·예약 확인서 중 하나를 OCR 인증으로 통과한 경우, 행과 펼침 영역에 "인증" 배지가 표시됩니다
- 배지 디자인: 초록색 작은 pill \(8x16px\) \+ 체크 아이콘 \+ "인증" 텍스트 — 작성자 닉네임 우측에 배치
- 미인증 게시글은 배지 없음 \(별도 표식 없이 깔끔하게\)
- 펼침 시 인증 게시글은 첨부된 인증 이미지 썸네일도 함께 표시 \(영수증·처방전 마스킹 처리됨\)
- 배지가 있는 게시글은 신고 처리 시 \[반려\] 우선 검토 \(신뢰도 높음\)

### 주요 액션 / 상태

__액션__

__설명__

\[삭제\]

게시글·댓글 선제 삭제 \(감사 로그 보존, 복원 가능\)\. 확인 모달 → 행은 회색 페이드, 상태 pill "삭제됨"

\[복원\]

삭제된 게시글·댓글 복원 \(30일 유예 기간 내\)\. 행 회색 해제, 사용자 앱에서 다시 노출

탭 전환

전체/게시글/댓글 필터링 \(게시글 탭에서는 댓글 숨김, 댓글 탭에서는 부모 게시글 함께 표시\)

### 운영 정책

- 기본 원칙: 신고 대응이 우선 \(선제 탐색·삭제 지양\)
- 선제 삭제가 정당한 케이스: 명백히 부적절\(욕설·광고·개인정보 노출\), 법적 분쟁 우려\(의료법 위반·명예훼손 명백\)
- 삭제 게시글은 감사 로그 보존, 복원 가능
- 의사 후기는 §3\.7 별도 화면에서 관리

## 3\.4 홈탭 운영

### 화면 목적

사용자 앱 홈 화면에 노출되는 콘텐츠\(배너·인기 명의·추천 게시글\)를 관리합니다\.

### 화면 구조 \- 서브탭 3종

- 3\.4\.1 상단 배너 — 캠페인·이벤트 배너 등록·노출 관리
- 3\.4\.2 주요 질환 인기 명의 — 질환별 추천 의사 큐레이션
- 3\.4\.3 추천 게시글 — 홈탭에 노출할 게시글 큐레이션

*서브탭은 화면 상단의 가로 탭 형태로 배치\. 한 번에 하나의 영역만 표시되며, 첫 진입 시 "상단 배너"가 기본 활성 상태입니다\. 다른 탭 클릭 시 즉시 전환 \(페이지 리로드 없음\)\.*

### 3\.4\.1 상단 배너

*화면 좌측에 배너 카드 목록, 우측에 사용자 앱 미리보기 패널 \(2단 그리드 레이아웃, 1fr · 320px\)\.*

### 화면 구성 — 좌측: 배너 카드 목록

- 헤더: "배너 목록 \(드래그로 순서 변경\)" 텍스트 \+ 우측 \[\+ 배너 추가\] 버튼
- 카드 목록: 각 배너마다 한 카드, 세로 배치
- 카드 구성 \(좌→우\):
	- • grip\-vertical 아이콘 \(드래그 핸들, 회색\)
	- • 썸네일 \(80x48px, 배너 이미지 또는 색상 그라데이션\)
	- • 메인 영역: 배너 제목\(14px 굵게\) \+ 노출 기간·누적 클릭수 \(11\.5px 회색\)
	- • 노출 토글 \("노출" 라벨 \+ 9x5 스위치, ON 시 파랑·OFF 시 회색\)
	- • \[수정\] 버튼 \(편집 모달 진입\)
- 예약 상태 카드: opacity 60% \+ "예약: 05\.20~05\.31" 표시 \+ 토글 자동 OFF

### \[\+ 배너 추가\] 모달 \(배너 작성\)

- 트리거: 상단 액션 바 \[\+ 배너 추가\] 버튼
- 모달 크기: max\-width 560px · width 90vw
- 헤더: "배너 작성" 제목 \+ \[×\] 닫기
- 필드 5종:
	- • 이미지 업로드: 드래그앤드롭 영역 \+ 클릭 파일 선택\. 권장 크기 720x320, JPG/PNG, 2MB 이하
	- • 제목: 텍스트 입력 \(최대 30자\)
	- • 노출 기간: 시작 날짜·종료 날짜 두 input \(YYYY\-MM\-DD\)
	- • 링크 URL \(선택\): 클릭 시 이동할 URL\. 미입력 시 클릭 동작 없음
	- • 초기 노출 상태: 라디오 \(즉시 노출 / 노출 OFF로 저장\)
- 푸터: \[취소\] / \[저장\] 버튼
- 동작: \[저장\] 클릭 → 모달 닫힘 \+ 새 카드 목록 맨 아래 추가 \+ 토스트 "배너가 추가되었습니다"

### \[수정\] 모달 \(배너 수정\)

- 기존 배너 카드의 \[수정\] 버튼 클릭 → 추가 모달과 동일 구조
- 기존 값으로 사전 채움\. 푸터에 \[삭제\] 버튼 추가 \(좌측 정렬, 빨간색\)
- \[삭제\] 클릭 → 확인 모달 → 완전 삭제 \(감사 로그 보존\)

### 화면 구성 — 우측: 사용자 앱 미리보기 패널

- "사용자 앱 미리보기" 라벨 \+ 모바일 프레임 \(140x240px\)
- 현재 노출 중\(토글 ON \+ 기간 내\) 배너만 표시\. 슬라이드 인디케이터\(점\) 표시
- 미리보기 새로고침: 카드 변경 시 즉시 반영

### 운영 정책

- 동시 노출 배너 권장: 2~3개
- 캠페인 종료 후 즉시 \[노출 OFF\] 또는 \[삭제\] 처리
- 드래그 순서: 위쪽 배너가 사용자 앱 슬라이드에서 먼저 노출

### 3\.4\.2 주요 질환 인기 명의

*질환별로 카드를 만들고, 각 카드 안에 의사 칩을 추가하는 2단계 구조\. 사용자 앱에서는 카드별로 슬라이드 노출\.*

### 화면 구성 — 상단 액션 바

- 좌측 텍스트: "현재 홈탭에 노출 중인 질환: N개 \(전체 질환 마스터: 100여 개\)"
- 우측 버튼: \[\+ 질환 추가\] \(파란색\)

### 화면 구성 — 질환 카드 \(각 카드\)

- 카드 헤더 \(좌측\): 질환 아이콘 \(40x40, 분류 색상 배경\) \+ 질환명\(14px 굵게\) \+ "홈탭 노출 N명" 텍스트
- 카드 헤더 \(우측\):
	- • "정렬" 라벨 \+ \[수동 / 자동\] 토글 버튼
	- • \[🗑\] 휴지통 아이콘 버튼 \(빨간색, title="질환을 홈탭에서 제거"\)
- 카드 본문: 의사 칩 가로 배치 \(flex\-wrap\)
- 의사 칩 구성:
	- • 인증 의사: 파란 배경\(blue\-50\) \+ 체크 배지 아이콘 \+ 의사명 \+ 병원명 \+ \[×\] 제거
	- • 미인증 의사: 회색 배경\(gray\-50\) \+ 배지 아이콘 없음 \+ 의사명 \+ 병원명 \+ \[×\] 제거
	- • 점선 테두리 \[\+ 의사 추가\] 버튼 \(의사 칩 끝에 배치\)
- 빈 카드 \(의사 0명\): "등록된 의사가 없습니다" 안내 \+ \[\+ 의사 추가\] 버튼

### \[\+ 질환 추가\] 모달

- 트리거: 상단 액션 바 \[\+ 질환 추가\] 버튼
- 모달 크기: max\-width 560px · width 90vw
- 헤더: plus\-circle 아이콘 \+ "홈탭에 질환 추가" 제목 \+ \[×\] 닫기
- 본문:
	- • "질환 마스터에서 검색" 라벨
	- • 검색창: 돋보기 아이콘 \+ "질환명 검색 \(예: 당뇨, 천식, 디스크\)" placeholder \+ autofocus
	- • 결과 리스트 \(max\-height 360px, 스크롤\): 매칭 질환 항목 — 질환명 \+ 분류 \+ \[추가\] 버튼
	- • 이미 노출 중인 질환은 \[노출 중\] 배지 \+ 비활성 표시
	- • 검색어 없을 때: 전체 질환 마스터 표시 \(인기순\)
	- • 하단 안내: "추가하면 빈 카드로 등록되고, 의사를 별도로 추가해야 노출됩니다\."
- 푸터: \[취소\] 버튼만 \(추가는 항목별 \[추가\] 버튼으로 즉시 실행\)
- 동작: 항목별 \[추가\] 클릭 → 모달 닫힘 \+ 빈 질환 카드가 목록에 추가 \+ 토스트 "N 질환이 홈탭에 추가되었습니다\. 의사를 추가해주세요\."

### \[\+ 의사 추가\] 모달 \(각 질환 카드 내\)

- 트리거: 질환 카드 내의 점선 \[\+ 의사 추가\] 버튼
- 모달 크기: max\-width 640px · width 90vw
- 헤더: user\-plus 아이콘 \+ "의사 추가" 제목 \+ 부제 "OO 질환의 추천 의사로 추가" \+ \[×\] 닫기
- 본문:
	- • "의사 마스터에서 검색" 라벨
	- • 검색창: 돋보기 아이콘 \+ "의사명·소속 병원 검색 \(예: 김재훈, 서울대\)" placeholder \+ autofocus
	- • 결과 리스트 \(max\-height 360px, 스크롤\): 의사 항목 — 체크박스 \+ 의사명 \+ \(인증 시 ✓ 배지\) \+ 전문과 \+ 병원명 \+ 우측 평점\(⭐ 4\.8\)·후기 수
	- • 검색 키워드 매칭: 의사명·전문과·병원명 모두
	- • 다중 선택 가능 \(체크박스\)
	- • 하단 안내: "인증된 의사는 사용자 앱에서 ✓ 배지로 표시됩니다\. 권장: 질환당 4~6명\."
- 푸터: 좌측 "선택: N명" 카운터 \+ 우측 \[취소\]/\[추가\] 버튼
- 동작: \[추가\] 클릭 → 모달 닫힘 \+ 선택된 의사 칩들이 질환 카드에 추가 \+ 토스트 "OO 질환에 의사 N명이 추가되었습니다"
- 예외: 선택 0명 \+ \[추가\] 클릭 → 토스트 "추가할 의사를 선택해주세요" \(모달 유지\)

### 의사 칩 제거 \(\[×\] 버튼\)

- 트리거: 의사 칩 내부 \[×\] 버튼 클릭
- 동작: 확인 모달 없이 즉시 칩 제거 \+ 토스트 "의사가 제거되었습니다"
- 의사 마스터는 유지 — 다른 질환·검색 결과에는 그대로 노출

### 질환 제거 \(\[🗑\] 버튼\)

- 트리거: 질환 카드 헤더의 \[🗑\] 휴지통 아이콘
- 확인 모달: "홈탭에서 'OO' 질환을 제거합니다" \+ 노란 안내 "질환 마스터와 의사 데이터는 그대로 유지됩니다"
- 푸터: \[취소\]/\[제거\] \(빨간색\)
- 동작: 카드가 페이드 아웃 \+ 노출 카운트 \-1 \+ 토스트

### 정렬 모드 토글 \(\[수동\] / \[자동\]\)

- 수동: 질환 카드 안의 의사 칩을 드래그로 직접 순서 변경
- 자동: 시스템이 평점·후기 수 기반으로 산출\. 가중치 = 평점 60% \+ 후기 수 25% \+ 최근 활동 15% \(개발팀 영역\)
- 자동 모드에서는 \[\+ 의사 추가\] 버튼 비노출 — 시스템이 자동 선정하므로 수동 추가 불가\. 수동 보강이 필요하면 정렬을 \[수동\]으로 전환 후 추가 \(추후 고도화 시 자동 모드에서도 수동 보강 기능 검토 가능\)
- 자동 모드로 전환 시 "수동 정렬이 자동 정렬로 대체됩니다" 확인 모달 → \[확인\] 시 의사 순서 자동 재배치

### 운영 정책

- 질환 마스터는 사용자 앱 사용자가 게시글 작성 시 자동 증가 \(개발팀 영역\)
- 권장: 홈탭 노출 질환 4~6개, 각 질환당 추천 의사 4~6명
- 의사 인증 여부: 인증 의사 우선 노출이 사용자 신뢰도 향상에 유리

### 3\.4\.3 추천 게시글

*커뮤니티 게시글 중 홈탭에 노출할 게시글을 수동 큐레이션\. 사용자 앱 홈에서 카드 슬라이드 형태로 노출\.*

### 화면 구성 — 상단 액션 바

- 좌측 텍스트: "현재 추천 게시글: N개 \(추천 슬롯 권장 4~6개\)"
- 우측 버튼: \[게시글 검색 추가\] \(파란색\)

### 화면 구성 — 추천 게시글 테이블

- 컬럼: 순서\(드래그 핸들\) / 제목 / 작성자 / 좋아요 / 댓글 / 처리
- 각 행 구성:
	- • grip 아이콘 \(드래그 순서 변경\)
	- • 제목 \+ 작성일
	- • 작성자 닉네임 \(인증 사용자는 ✓ 배지\)
	- • 좋아요 수 / 댓글 수 \(tabular\-nums 우측 정렬\)
	- • \[제거\] 버튼 \(빨간색 작은 버튼\)

### \[게시글 검색 추가\] 모달

- 트리거: 상단 액션 바 \[게시글 검색 추가\] 버튼
- 모달 크기: max\-width 720px · width 90vw
- 헤더: "추천 게시글 추가" 제목 \+ \[×\] 닫기
- 본문:
	- • 검색창: "제목·내용·작성자 검색" placeholder \+ autofocus

## 3\.8 의사 관리

### 화면 목적

### 화면 구성

- 탭: 전체 / 정상 / 취소됨
- 검색: 이름·소속·면허번호
- 컬럼: 의사명 / 소속 병원 / 면허번호 / 인증 상태\(pill\) / 대표 전화 / 노출 여부\(토글 ON·OFF, 클릭 시 즉시 변경\) / \[편집\] 버튼
- 인증 상태 pill: 인증됨\(파랑\) / 취소됨\(빨강\)
- 취소된 의사 행: 전체 회색\(opacity 낮음\) \+ \[편집\] 버튼 비노출

### \[편집\] 모달 구조 \(5개 섹션\)

- 트리거: 의사 행 우측 \[편집\] 버튼 \(취소된 의사는 버튼 없음\)
- 모달 크기: max\-width 720px · width 90vw · 본문 max\-height 70vh 스크롤
- 헤더: stethoscope 아이콘 \+ "의사 정보 편집" \+ \[×\] 닫기
- 상단 안내 박스 \(파란 배경\): "면허·소속 정보는 개발팀 DB 직접 변경 영역\. 평점·후기 수는 시스템 자동 산출\. 본 화면에서는 인증 상태와 운영 데이터만 관리"
- 섹션 1 — 의사 식별 정보 \(🔒 읽기 전용, 회색 배경 박스\):
	- • 의사명 / 전문과 / 소속 병원 / 진료과 / 면허번호 — 자물쇠 아이콘 \+ "편집 불가 — 개발팀 영역" 명시
- 섹션 2 — 면허 인증 상태:
	- • 인증됨: 파란 배경 \+ badge\-check 아이콘 \+ 인증 완료일 \+ \[인증 취소\] 버튼\(빨간색\) \+ ℹ "인증 취소는 단방향\. 재인증은 개발팀 영역"
	- • 취소됨: 빨간 배경 \+ x\-circle 아이콘 \+ 취소일 \+ 취소 사유 \+ ℹ "취소된 의사는 사용자 앱에 노출되지 않으며 후기·댓글 권한 없음\. 재인증은 개발팀 영역"
- 섹션 3 — 연락처:
	- • 대표 전화 / 온라인 예약 URL \(font\-mono, "\(클릭 시 새 창으로 열림\)" 안내\)
- 섹션 4 — 소개 요약: 대표 경력\(한 줄, 텍스트 input\) / 대표 학력\(한 줄, 텍스트 input\)\. 각 필드는 단일 라인으로 사용자 앱 프로필 카드에 표시되는 요약 문구
- 섹션 5 — 상세 리스트: 경력 전체 / 학력 전체 / 논문·학술 활동 \(각 textarea, 한 줄에 항목 하나씩\) / 세부 전공 분야 \(텍스트 input, 쉼표로 구분\)\. 최대 노출 줄 수 등 표시 정책은 사용자 앱 프론트엔드 정의를 따름
- 섹션 6 — 노출 설정: 체크박스 ON/OFF 토글 방식\. ON 시 사용자 앱 검색·추천에 노출 / OFF 시 비노출 \(이미 저장된 후기·게시글은 그대로 유지\)\. 기본값 ON\. 인증 취소된 의사는 노출 설정 여부와 무관하게 자동 비노출
- 푸터: \[취소\] / \[저장\] 버튼\. 동작: 저장 클릭 → 토스트 "OOO 의사 정보가 저장되었습니다"

### \[인증 취소\] 확인 모달

- 트리거: 편집 모달 내 \[인증 취소\] 버튼
- 필드: "취소 사유" textarea \(필수\) \+ ℹ "인증 취소는 단방향\. 재인증은 개발팀에 요청하세요\."
- 푸터: \[돌아가기\] / \[인증 취소 확인\] \(빨간색\)
- 동작: 확인 → 편집 모달의 인증 상태 섹션 즉시 "취소됨"으로 변경 \+ 의사 행 pill "취소됨"\(빨강\) \+ 토스트

### 운영 정책

- 개발팀 DB 직접 변경 영역: 신규 등록·삭제·식별 정보 변경\(이름·전문과·소속·진료과·면허번호\), 인증 취소된 의사의 재인증
- 어드민 편집 가능 영역: 연락처, 소개, 상세, 노출 설정, 인증 취소
- 평점·후기 수는 시스템 자동 산출 \(어드민 수정 불가\)

## 3\.10 사용자 관리

### 화면 목적

회원 검색 및 활동 이력을 조회합니다\. 베타에서는 조회 전용 \(회원 제재 액션은 정식 런칭 시 도입\)\.

### 화면 구성

- 검색: 닉네임·이메일
- 컬럼: 닉네임\(의사 인증 사용자는 녹색 \[✓ 의사\] 배지\) / 이메일\(마스킹\) / 가입일 / 신고 누적 / 병원 진료 인증 / 상태 pill / 화살표\(→\)

## 3\.11 운영 정책 설정

### 화면 목적

사용자 앱에 적용되는 운영 파라미터를 코드 배포 없이 변경합니다\. 베타 버전의 기본 항목은 콘텐츠 수정·삭제 가능 기간\(N일\) 3개 그룹\(게시글·댓글·리뷰\)입니다\. 정책 본문\(SSOT\)은 외주 패키지 문서 「AIGA 사용자 등급별 정책 화면정의서」 §5\-0 「수정·삭제 기간 운영 정책」을 따릅니다\.

### 화면 구성

- 섹션 1 — 수정·삭제 기간: 3개 그룹 카드 \(게시글 그룹 / 댓글 그룹 / 리뷰 그룹\)\. 각 카드는 모드 선택\(수정 불가 / N일 / 무제한\) \+ N일 모드일 때만 활성화되는 일수 입력 필드\(1 이상 자연수\) \+ 현재 적용값 표시 \+ \[저장\] 버튼
- 섹션 2 — 변경 이력: 본 화면에 최근 20건 테이블 표시 \(변경 시각 / 운영자 / 항목 / 이전값 → 변경값\)\. 전체 이력은 §1\.4에 따라 백엔드 감사 로그에 자동 기록되며, 어드민 조회 UI는 베타 미포함\(§1\.4 감사 로그 운영 정책 참조\)\. 필요 시 개발팀 DB 추출로 대응
- 각 카드 ℹ 안내 문구: "수정 불가 = 작성 직후부터 수정·삭제 차단, 무제한 = 작성 시각과 무관하게 항상 가능\. 저장 즉시 사용자 앱에 반영되며 기존 콘텐츠에도 소급 적용됩니다\."

### \[저장\] 동작

- 설정 변경 후 \[저장\] 클릭 → 확인 모달: "\[그룹명\] 수정·삭제 정책을 '\[이전 설정\]' → '\[새 설정\]'으로 변경합니다\. 사용자 앱에 즉시 반영되며, 작성된 지 오래된 콘텐츠도 새 기준이 소급 적용됩니다\. 진행할까요?" \(설정 표기 예: '7일' / '무제한' / '수정 불가'\)
- \[확인\] → 즉시 저장 \+ 사용자 앱 캐시 무효화 \+ 변경 이력 1건 추가 \+ 토스트 "저장되었습니다"
- \[취소\] → 입력값 원복, 저장 없음

### 입력 검증

- N일 모드일 때 일수는 1 이상 자연수만 \(0·음수·소수·문자 입력 시 \[저장\] 비활성\)\. 수정 불가·무제한은 모드 선택만으로 적용되며 일수 입력 불필요
- 현재 적용값\(모드 \+ 일수\)과 동일하면 \[저장\] 비활성
- 상한 권장값: 365일 \(그 이상도 기술적으로 가능하나 운영상 의미가 약함\)\. 초과 입력 시 경고 토스트 노출 후 저장은 허용

### 운영 정책

- 최초 기본값은 발주사 결정\. 외주사는 어드민 UI에서 변경 가능하도록 설계\(하드코딩 금지\)
- 3개 그룹 각각 독립 설정: ① 게시글 그룹\(수정·삭제\), ② 댓글 그룹\(댓글·대댓글 수정·삭제\), ③ 리뷰 그룹\(수정·삭제\)
- 적용 기준: 수정 불가 = 작성 직후부터 차단 / N일 = 작성 시각부터 N일 이내만 가능 / 무제한 = 작성 시각과 무관하게 항상 가능 \(정책 본문은 §5\-0 참조\)
- 변경 권한은 모든 운영자\(베타에서는 §1\.4에 따라 단일 어드민 계정\)에 부여\. 모든 변경은 감사 로그에 자동 기록\(§1\.4\)
- 인증 정책 상호작용: N일을 줄이면 과거 인증된 게시글의 재인증 접근이 소급적으로 차단될 수 있음\. 변경 전 확인 모달에 안내 문구 추가 권장
- 확장성: 운영 중 추가로 어드민에서 변경 가능한 파라미터가 생기면 본 화면에 섹션으로 추가\(예: 콘텐츠 안전 정책의 카테고리별 안내 문구 매핑\)
